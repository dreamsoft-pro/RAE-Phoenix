Co jeszcze pomoże agentom działać precyzyjniej (ponad Twój plan)
#ŹRÓDŁO→DOWÓD
Każda metadana powinna mieć wskazanie skąd się wzięła (source-of-truth). Dorzuć do payloadów pola:
evidence: {file, start_line, end_line, rule} – dzięki temu agent wie, dlaczego dany endpoint/route/wywołanie zostało wykryte (łatwo weryfikować i odszumiać).
confidence: liczbowy confidence (0–1) dla poszczególnych wykryć (np. endpoint z literalem = 0.95, heuristic z template literal = 0.7).
#RELACJE→KONTRAKTY
Oprócz grafu relacji dodaj kontrakty:
io_contract: wejścia/wyjścia funkcji/serwisów w prostym schemacie (np. params, returns, throws).
api_contract: {method, path, queryKeys, bodyKeys} wyciągane heurystycznie z wywołań $http.
#RANKING KRYTYCZNOŚCI
Agent działa szybciej, gdy ma ranking „gdzie zacząć”:
criticality_score: funkcja f(usage_degree, route_exposure, complexity, churn, errors);
usage_degree – ile razy dany symbol jest wywoływany;
route_exposure – czy dotyczy głównych ścieżek (np. /checkout);
churn – ile zmian w ostatnich N commitach (z git log --follow);
errors – wzmianki fix: w commitach dot. pliku/symbolu.
#WIELOWARSTWOWE WYSZUKIWANIE
Aby agent mógł łączyć „co, gdzie, jak”, indeksuj do oddzielnych kolekcji:
code_chunks (dense + sparse),
route_maps (mapy URL↔komponent/ctrl),
api_calls (wywołania API: endpoint↔symbol↔plik↔linia),
domain_glossary (tagi biznesowe, definicje),
call_graph (krawędzie caller→callee z wagą i confidence).
#HEURYSTYCZNE STRESZCZENIA (bez LLM)
Dla szybkości/bezpieczeństwa możesz dodać regułowe summary_en (1-linijkowe) zanim włączysz LLM:
baza na kind, business_tags, nazwie symbolu i najczęstszych czasownikach w ciele funkcji.
#REŻIM JAKOŚCI DANYCH
Zanim agent ruszy, przepuść KB przez walidator:
sprawdź unikalność id, brak NULI w kluczowych polach, zakresy start/end, kompletność filePath (istnieje w repo), i minimalny zestaw metadanych.
Gotowe pliki, które dla Ciebie przygotowałem
Dostajesz drop-in indexer (Node ESM) i enricher blame (Python).
Możesz je od razu pobrać i wrzucić do repo (ścieżki sugerowane poniżej).
scripts/js_html_indexer.mjs
ESM indexer AngularJS 1.x → JSONL z metadanymi:
calls_functions, api_endpoints, ui_routes,
prosty cyclomatic_complexity,
heurystyczne business_tags,
route chunks z $stateProvider/$routeProvider.
scripts/enrich_git_blame.py
Dokleja do każdego chunka metadata.git_last_commit na podstawie git blame dla zakresu linii.
README_INDEXER_ADDONS.md
Krótka instrukcja uruchomienia i wymagane zależności.
Wymagane zależności (Node)
npm i -D fast-glob @babel/parser @babel/traverse
Jak użyć (krok po kroku)
# 1) Skopiuj pliki do repo (sugerowane ścieżki)
cp /mnt/data/feniks_addons/scripts/js_html_indexer.mjs ./scripts/
cp /mnt/data/feniks_addons/scripts/enrich_git_blame.py ./scripts/
cp /mnt/data/feniks_addons/README_INDEXER_ADDONS.md ./docs/README_INDEXER_ADDONS.md
# 2) Zainstaluj zależności do indexera
npm i -D fast-glob @babel/parser @babel/traverse
# 3) Zbuduj JSONL z rozszerzonymi metadanymi
node scripts/js_html_indexer.mjs --root ./frontend-master --out runs/latest/chunks.mjs.jsonl
# 4) Wzbogacenie o git blame
python3 scripts/enrich_git_blame.py --repo . --in runs/latest/chunks.mjs.jsonl --out runs/latest/chunks.enriched.jsonl

Dalej możesz wpiąć to w swój pipeline (Python) — wczytaj chunks.enriched.jsonl i zmapuj do Chunk (masz już pola metadata.* zgodne z Twoimi założeniami).
Quick diff vs. Twój Plan — co już realizuje się kodem
Jawny graf wywołań: zbieramy calls_functions z AST (MemberExpression / Identifier).
Mapowanie endpointów: wykrywamy $http.*(url) i $resource(...) → metadata.api_endpoints.
Powiązania trasa↔komponent: z $stateProvider.state i $routeProvider.when; dopinamy ui_routes do pasujących kontrolerów i generujemy osobne route-chunki.
Kontekst biznesowy: heurystyczne business_tags; miejsce na summary_en.
Historia i „ryzyko”: git_last_commit (hash/author/date/message) przez enrich_git_blame.py.
Złożoność: prosty licznik decyzji (cyclomatic_complexity) — wystarczający do rankingu kandydatów do refaktoryzacji.
Co warto dołożyć „następny krok” (łatwo)
call_graph kolekcja: podczas traversalu zapisuj krawędzie caller→callee (zliczaj).
api_calls kolekcja: osobny JSONL z rekordem {file, line, method, url, symbol} dla precyzyjnych zapytań.
confidence & evidence: do metadata wpisz {source:'ast', rule:'$http.literal', confidence:0.95}; każdy atrybut może mieć osobny evidence-list.
LLM summaries (opcjonalnie): zrób osobny krok, który generuje summary_en tylko dla chunków z wysokim criticality_score, by oszczędzać budżet.
Integracja z Twoim pipeline (przykład)
Node indexer → runs/latest/chunks.mjs.jsonl
Python blame enricher → runs/latest/chunks.enriched.jsonl
Python core (Twój load_chunks_from_ast) czyta enriched JSONL, mapuje pola do Chunk.metadata.
Embeddings (dense+tfidf) → Qdrant (code_chunks).
(opcjonalnie) osobne kolekcje: route_maps, api_calls, call_graph — załaduj z tych samych artefaktów.
Jeśli chcesz, dorzucę patch do Twojego Pythona (types.py, mapper JSONL→Chunk, walidator metadanych) i gotowe komendy commit/push, ale już teraz masz komplet narzędzi, które od ręki realizują większość Twojego planu i kładą „dobre fundamenty” pod precyzyjną pracę agentów.
